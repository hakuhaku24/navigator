# 5/6 老師講評暨組員討論

## 教授回饋與建議統整

根據提供的對話內容，教授針對旅遊行程規劃系統提出了以下核心建議與評估，主要涵蓋技術優化、商業模式及行政庶務三個面向：

### 1. 資料架構與提示詞工程（Prompt Engineering）優化

- **避免全量資料輸入：** 教授指出，若系統庫存有一萬個景點，直接將所有景點資料全數塞入提示詞（Prompt）中會極度缺乏效率，且大語言模型（LLM）未必能精準檢索到所需資訊。
- **建立外部索引與動態載入機制：** 系統應將資料解耦。將 LLM 已經具備的常識，與系統獨有的資料（如 10% LLM 查不到的特殊景點、5% 的自訂索引）分離。在執行腳本時，先讀取使用者需求，再從外部文件（如 Markdown 檔）抓取對應的景點資訊注入提示詞中。這能大幅縮小檢索範圍、降低 Token 消耗並提高準確度。
- **利用使用者偏好做精準觸發：** 建議將景點進行分級。利用使用者的特定喜好作為「密碼」或觸發條件，當條件符合時，系統才調用特定的提示詞區塊與附加資料。這能使產出的行程具備高度客製化，取代無差異化的大量資料輸入。

### 2. 商業價值與系統定位（Business Model）(這個只是建議)

- **確立核心競爭力：** 教授強調系統必須具備獨特的商業價值，需思考「為何使用者不直接詢問一般 LLM，而要使用此系統？」，避免專案僅停留在純學術實作的層次。
- **B2B 商業應用（旅行社端）：** 系統架構可轉化為旅遊業者（如 KKday）的內部生成工具，用來快速自動打包特定的旅遊產品（例如：從關西機場出發的特定神社一日遊票券），提升上架效率。
- **B2C 商業應用（利基市場端）：** 針對有特殊偏好的深度玩家（例如重度登山客或特定主題愛好者），提供極具專業度與針對性的行程規劃。透過提供「差異化內容」，系統可向這類特定受眾收取小額費用（例如單次或訂閱制），創造實質收益。

### 3. 技術實作與評估

- **搜尋引擎 API 替換：** 教授已知悉團隊因 Google Custom Search API 停用或限制，而將搜尋功能改用 DuckDuckGo 作為備案的做法，並同意此調整。
- **系統驗證機制：** 團隊目前將驗證架構分為三大區塊，教授對於其中將部分景點設為「可替換（L2）」的動態邏輯表示理解，並認同目前的實作方向可先進行測試與推進。

### 4. 行政庶務與經費核銷

- **補助額度與報帳：** 針對團隊目前 API Token 約 9.9 台幣的消耗，教授提醒該專題每組有 2,000 元新台幣的經費補助。
- **請示系辦核銷規定：** 由於 API 費用涉及國外信用卡扣款，教授明確指示團隊需派員向「系辦」確認具體的報帳流程細節（包含收據形式、是否需要統編等），以確保後續能順利核銷款項。
