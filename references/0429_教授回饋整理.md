# Navigator（領航者）4/29 教授回饋錄音整理 第一版

這份錄音記錄了你們針對「領航者」旅遊即時應變系統與教授進行的深入討論。教授的講評非常精闢，主要集中在**如何從「技術堆砌」轉向「商業價值」**，並強調了 AI 原生系統的邏輯準確性。
以下我為你們整理的詳細講評分析及下週行動建議：

## 📋 教授講評詳細整理

教授將你們的系統定位為**「AI 時代的旅行社」**，認為核心價值不在於技術多複雜，而是在於資訊的**「可信度」**與**「應變能力」**。

### 1. 四大功能模組的反饋

- **景點驗證 (POI Verification)：**
  - **核心價值：** 這是系統的「底層基礎」。教授非常認可透過多個 Agent（如部落格抓取 vs. 官方 API 交叉比對）來驗證景點是否仍在營業或其真實評價。
  - **建議：** 應區分「預運算 (Pre-compute)」與「即時查詢 (On-demand)」。針對特定地區（如大溪）可以預先驗證以節省 Token。
- **行程規劃 (OR 模型)：**
  - **警示：** 教授提醒「數學公式一旦列出就必須絕對正確」。
  - **邏輯優化：** 不要過度追求複雜的預算成本計算（因資料來源可能不準），重點應放在「行程的時間串接」與「多準則決策（評分、投票）」。
- **應變系統設計 (Contingency SOP)：**
  - **核心定位：** 這是你們與一般旅遊 App 的最大差異。
  - **重點：** 不要只給「一個完美的答案」，而是要提供「應變計畫 (Backup Plan)」。例如：下大雨時，除了原本的景點， Agent 應即時推薦附近的室內替代方案。
- **補充資料與預約：**
  - **實用性：** 串接 Overpass API 尋找廁所、ATM、飲水機是很好的痛點解決，但需注意資訊的時效性。

### 2. 系統邏輯與技術架構

- **AI 原生思維 (AI Native)：** 教授稱你們為「AI 原住民」，但也提醒不要為了自動化而自動化。
- **提示工程 (Prompt Engineering)：** 建議結合 **RAG (檢索增強生成)** 與 **Context Engineering**。不要只問 AI「下雨怎麼辦」，而是要給 AI 充分的背景資訊（如：目前在大溪、有 5 個人、下大雨、原本要去老街、附近有這些咖啡廳），讓它產出應變建議。
- **準確性勝過複雜度：** 旅遊系統一旦出錯（如帶用戶去一家倒閉的餐廳），用戶就會失去信任。

## 🚀 下週建議行動清單 (Next Steps)

教授希望下次看到的是**「具備實際數據的 Prototype」**，而非單純的架構圖。

### 第一階段：數據與驗證實作 (Data & Verification)

- **[ ] 完成「大溪」示範區的 POI 驗證：**
  - 針對大溪（或你們選擇的測試點）抓取 50 個景點，實際執行一次 Agent 交叉驗證邏輯。
  - 產出 JSON 格式的驗證報告，標註哪些資訊是「官方證實」、哪些是「部落客推薦」。
- **[ ] 建立應變情境清單：**
  - 窮舉至少 5 種突發狀況（大雨、景點臨時關閉、人潮爆滿、手機沒電、團員走散）的處理 SOP。

### 第二階段：提示工程優化 (Prompt Engineering)

- **[ ] 設計「應變 Agent」的 Prompt 範本：**
  - 實踐教授提到的 Context Engineering，測試在不同突發狀況下，AI 給出的「替代方案」是否合理且具備地理鄰近性。

### 第三階段：原型展示準備 (Prototype Prep)

- **[ ] 製作 Demo 案例：**
  - 不要只講技術細節，直接展示一個案例：_「原本規劃去 A 景點，突然下大雨，系統自動推播 B 咖啡廳（室內）並顯示有 5 個空位，用戶一鍵確認更換。」_
- **[ ] 修正 OR 模型公式：**
  - 簡化計算邏輯，確保公式中的權重（Weight）與準則（Criteria）有明確的業務邏輯支持，避免被教授問倒。
    **💡 小提醒：**
    教授最後提到：「你們的系統包山包海，選 2-3 個做得最好的模組深入開發即可。」建議你們下週集中火力把**「景點驗證」**與**「下雨應變」**這兩個場景做紮實。

# Navigator（領航者）4/29 教授回饋錄音整理 第二版

這份錄音記錄了你們針對「領航者」旅遊即時應變系統與教授進行的第四週專題進度討論。教授的回饋非常深入，將你們的技術架構與實際商業價值進行了連結。
以下是教授講評的詳細整理與分析：

### 一、 核心定位：AI 時代的「個人旅行社」

教授認為你們系統的最終目標是成為一個 **AI 時代的旅行社**。他強調你們這組具備「AI 原住民」的優勢，能夠自然地運用 AI 代理人（Agent）處理複雜邏輯，而不僅僅是當作加速工具。

### 二、 四大功能模組的深度建議

根據你們更新的架構圖，教授針對提升「商業價值」的四項功能提供了具體指引：

#### 1. 景點驗證 (Spot Verification)

- **驗證邏輯：** 教授認同透過多個 Agent（如 Researcher）進行交叉驗證，比對部落格、Google Places 與 OpenStreetMap (OSM) 的資訊。
- **成本與效率：** 建議區分「預運算 (Pre-compute)」與「隨選 (On-demand)」。例如針對熱門地區（如大溪）可預先驗證景點資訊，以節省 Token 成本並提高速度。
- **多維度驗證：** 驗證不應只有評分，還應包含季節性、氣候與營業時間的細節，這些是後續行程規劃的重要參數。

#### 2. 行程規劃與 OR 模型

- **模型精簡：** 教授提醒不要過度陷入複雜的數學公式（如準確到個位數的預算計算），因為底層數據若不準，精確的公式反而會造成誤差。
- **決策關鍵：** 重點應放在「行程的時間串接」與「多準則決策」（如交通便利性、體力消耗程度）。

#### 3. 即時應變系統 (Contingency Design) — **核心價值**

- **應變計畫 (Backup Plan)：** 教授指出這才是系統最迷人的地方。系統不應只給一個完美的方案，而是要根據突發狀況（如基隆突然下雨）自動產出應變 SOP。
- **系統韌性：** 即便 AI 給出的建議不完美，只要「應變計畫」的邏輯正確（例如推薦鄰近的室內景點），系統就不會讓使用者在現場卡住。

#### 4. 補充資料與痛點解決

- **精準抓取：** 透過 Overpass API 尋找廁所、ATM、飲水機等生理需求，是解決旅遊痛點的亮點。
- **功能聚焦：** 教授警告不要試圖「包山包海」，建議從中選出 2-3 個做得最出色的模組進行深度展示，避免資源分散。

### 三、 技術實作與邏輯優化

- **提示工程 (Prompt Engineering)：** 建議結合 RAG（檢索增強生成）或 **Context Engineering**。不要直接問 AI 問題，而是將驗證過的景點資料、天氣、人數、使用者偏好封裝成完整的「上下文」餵給 AI，讓它產出最精準的應變決策。
- **資訊透明度：** 系統應標註資訊的來源與時效（例如：此資訊為 3 年前的部落格或 5 分鐘前的氣象局 API），讓使用者決定信任程度。

### 四、 下週行動建議：從架構轉向原型 (Prototype)

教授最後的總結是希望看到你們開始動手實作 **Prototype**，以驗證 AI Agent 的協作是否如想像中美好：

1.  **實測 Agent 協作：** 實際讓 Agent 跑一次「驗證景點」到「生成應變計畫」的完整流程，看是否會產生幻覺或邏輯錯誤。
2.  **情境演練：** 準備 1-2 個極端的應變案例（如大雨、景點臨時關閉），展示 AI 如何透過「提示詞工程」產出合理的應變計畫。
3.  **架構收斂：** 根據教授的回饋，將 OR 模型與應變邏輯進行最後的簡化與收斂，確保下週報告時有具體的實作數據支持。
    **這次教授的回饋非常肯定你們的應變邏輯。你們打算先從哪一個突發情境（例如天氣或景點關閉）開始實作 Agent 的應變測試？**
