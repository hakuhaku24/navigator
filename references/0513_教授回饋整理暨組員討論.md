### 一、 系統商業定位與法律風險防範 (源自音檔 39)

教授並未提及 B2B 或 B2C 的框架，而是從「法律合約風險」的角度切入，要求重新定義系統的輸出價值與受眾。

- **避開旅行社定位以規避法律責任：** 教授強烈建議系統不要定位為「旅行社」或「旅遊行程規劃」。因為旅行社出團涉及實體合約，若 AI 規劃出錯或違反合約，將面臨賠款與法律責任。
- **轉向自助旅遊的諮詢檢索工具：** 系統應精確定位為「專業旅遊諮詢的服務提供者」，針對「單身/個人旅遊人」提供服務。在此定位下，系統提供的是一個「先檢驗過的資訊查核點」，由於沒有實體出團合約，即可規避法律風險，同時彰顯系統作為資訊聚合器的價值。

### 二、 系統架構評估與價值延展 (源自音檔 39)

教授套用了某套包含「七個面向」的專案評估標準，特別針對其中的「第六」與「第七」面向，檢視你們的「系統觀」與「通用性」。

- **面向六（規模化檢驗）：** 如果有大量使用者湧入使用該系統，系統會發生什麼改變？架構是否能承受並維持價值？
- **面向七（資本化檢驗）：** 如果獲得大量資金挹注，系統的價值是否會隨之飆升？教授指出，許多不良的專案即便投入再多資金也無法提升價值，但他認為你們目前的架構若將「通用性」做好，是有機會隨著資本投入而將價值往上拉升的。
- **系統通用性 (Generalizability)：** 教授提醒，面對全台甚至全世界數以萬計的景點，系統不能只針對少數情境寫死邏輯（Hardcoding），必須建立一套標準化的資料架構與處理程序（Data Architecture）。

### 三、 應變系統的容錯機制與未決狀態處理 (源自音檔 39)

針對你們以「老梅綠石槽下雨」切換至「白沙灣探索館」的應變 Demo，教授指出了系統在真實世界運作的盲區。

- **API 的現實限制：** 教授提醒，事件偵測「沒那麼美好」，在偏鄉或特定區域，API 未必能精準查出即時降雨機率或其他動態資訊。
- **[✅ 已解決] 處理「未決 (Unresolved)」狀態：** `agents/contingency-handler/src/generators/llm-client.ts` 實作 Gemini 2.5 Flash → Claude Haiku → 空字串三層 fallback；`venue-detector.ts` 在 Google Places 不可用時回退到 POI 快照資料。LLM 掛掉時系統仍能回傳規則層的 ContingencyPlan，不會直接崩潰。景點聯絡電話 / 線下備用資訊尚未完整補入資料集（待後續補齊）。

### 四、 資料管線、RAG 與 Tool Use (源自音檔 34 & 39)

教授對你們目前的 Tool Use（工具調用）掌握度給予高度評價，但針對資料索引與上下文工程提出了具體改進方向。

- **[✅ 已解決] 搜尋引擎備援機制：** `agents/poi-verifier/src/validators/blog-search.ts` 實作 DuckDuckGo（免費，主要）→ Serper（付費，fallback，DDG < 2 筆時才觸發）。策略已符合教授確認的方向。
- **[✅ 已解決] 靜態與動態資料分流：** `poi_catalog` Supabase 表存靜態預處理資料（blog_snippets、pgvector embedding）；即時天氣仍走 CWA API 動態查詢。pgvector `match_poi_catalog` RPC 已建立（commit 80c78e9），可做語意相近景點搜尋而非每次重跑 API。
- **[✅ 已解決] Context/Harness Engineering：** `agents/contingency-handler` 五階段管線在送入 LLM 前，先用規則層（EV 計算 + 嚴格篩選 + 多準則排序）壓縮候選清單至 top-5，再將結構化結果注入 Prompt，避免塞爆上下文。MCP 整合尚未實作（可列為後續）。

### 五、 專案經費現實與開發策略 (源自音檔 39)

教授分享了系上的經費現況，並給予務實的開發建議。

- **經費落差：** 教授在系務會議中爭取提升 AI 專案的補助，因為資管系軟體專案通常只能分到約 2,000 元的補助，對比 IoT 硬體設備動輒 30 萬元的經費，有顯著落差。
- **窮人的方法：** 教授舉新加坡每月補助每人約 3,000 元推動 AI 使用的例子，勉勵你們在目前經費受限的狀況下，要試著用「窮人的方法」（如善用免費 API 額度、優化檢索邏輯降低 Token 消耗）來把系統的商業價值拉高。
